# JS 的执行

## 单线程执行

- 单线程执行：同一时间只能执行一个任务，只有把当前任务完成，才能执行下一个任务
这会造成**线程阻塞**（当前任务耗时过长，导致后面的任务不能被执行）

```js
for (let i = 0; i < 5000; i++) {
    console.log("一个耗时很长的同步任务");
}
console.log("后面的同步任务");
```

上例就出现了线程阻塞：`for` 循环执行完后，才会执行后面的代码

## JS 的线程组成

- 虽然浏览器是单线程执行 JS 代码的，但浏览器是以多线程协助 来实现单线程异步模型的，具体线程组成如下：
  ① GUI 渲染线程、② JS 引擎线程、③ 事件触发线程、④ 定时器触发线程、⑤ http 请求线程...
- 之所以说 JS 是单线程语言，是因为在 JS 代码运行的过程中，同一时间只存在⼀个线程在执行程序
  这里的同步、异步就是靠多线程切换的形式来实现的
- 我们常将上面的线程归纳为下列 2 条线程：
  ① 主线程：⽤来渲染页面、执行 JS 代码、触发事件...
  ② ⼯作线程：在幕后⼯作，执行异步操作，实现非阻塞的运行模式

# 同步 & 异步

## 同步任务 & 异步任务

- 同步任务：即 “非耗时程序”，都是在 [主线程] 执行；执行顺序：从上往下
- 异步任务：即 “耗时程序” (**事件**、**Ajax**、**定时器**等)，由 JS 委托宿主环境 (Node / 浏览器) 执行
- JS 的执行顺序遵循单线程的异步模型：**同步在前，异步在后**
- 根据 JS 单线程异步模型的规则，异步操作得通过 [回调函数] 实现
  ∴ 异步任务一定有回调函数！但请注意，有回调函数的不一定是异步任务！

```js
setTimeout(function () {
    console.log("异步任务");
}, 0);

[1].forEach(item => {
    console.log("同步任务");
});
```

在 [主线程] 中，JS 代码会被**从上往下**地一行行执行；遇到异步任务，则委托给宿主环境开辟新线程执行；
异步任务执行完后，会将其 [回调函数] 放入任务队列 Event Queue 等待；
等主线程将同步任务都执行完，就从 Event Queue 中获取最先入队的回调函数执行

```js
setTimeout(() => {
    console.log("异步任务");
}, 0);
for (let i = 0; i < 5000; i++) {
    console.log("一个耗时很长的同步任务");
}
// for 循环执行完后，才会执行 setTimeout 里面的代码
```

> #### 如果出现多个异步操作：

在异步操作完成时间**不同**的情况下，先完成的异步操作会先进入 Event Queue：
下例输出顺序：`sync` - `setTimeout 2` - `setTimeout 1`

```js
setTimeout(function () { // 这个耗时 2s
    console.log('setTimeout 1');
}, 2000);

setTimeout(function () { // 这个耗时 1s
    console.log('setTimeout 2');
}, 1000);

console.log("sync");
```

在异步操作完成时间**相同**的情况下，先**注册**的异步操作会先进入 Event Queue：
下例输出顺序：`sync` - `setTimeout 1` - `setTimeout 2`

```js
setTimeout(function () { // 前面的代码先注册
    console.log('setTimeout 1');
}, 2000);

setTimeout(function () { // 后面的代码后注册
    console.log('setTimeout 2');
}, 2000);

console.log("sync");
```

任务队列前面的异步操作被阻塞时，后面的异步操作也会被阻塞：
下例的输出顺序：`sync` - `setTimeout 1 start` - `setTimeout 1 end` - `setTimeout 2`

```js
setTimeout(function () { // 先进入执行栈
    console.log('setTimeout 1 start');
    for (let i = 0; i < 5000; i++) { // 被阻塞了
        console.log("一个耗时很长的同步任务");
    }
    console.log('setTimeout 1 end');
}, 1000);

setTimeout(function () { // 后面的异步操作也会被阻塞
    console.log('setTimeout 2');
}, 1000);
console.log("sync");
```

## 宏任务 & 微任务

> #### 宏任务

宏任务是 JS 中最原始的异步任务，包括：定时器、Ajax... 在主线程中按照同步代码的顺序，逐个进入工作线程挂起，再按照异步任务到达的时间节点，逐个进⼊异步任务队列，最终按照队列中的顺序进入执行栈执行

常见的宏任务：① **输入输出**、② **定时器**、③ **Ajax**、④ **事件**、⑤ **`requestAnimationFrame`**(浏览器) ... 
有些地方会说 UI Rendering 也是宏任务，但这是和微任务平行的操作，应该算是微任务；
`requestAnimationFrame` 是页面重绘前执行的操作（⑦ **重绘** 也是⼀个宏任务）

> #### 微任务

微任务是随着 ECMA 标准升级提出的新的异步任务。每⼀个宏任务执行前，程序会先检测，是否有**当次事件循环**未执行的微任务，优先清空本次的微任务后，再执行下一个宏任务；
每⼀个宏任务内部可注册当次任务的 [微任务队列]，在下一个宏任务执行前清空；
微任务也是按照进入任务队列的顺序进入执行栈的

常见的微任务：① **`process.nextTick`**(node.js)、② **`MutationObserver`**(浏览器)、③ **`Promise实例.then catch finally`**...

## 事件循环

[主线程] 会从上往下一行一行地执行 JS 代码；并注册本次循环的微任务和宏任务
遇到异步操作，则委托给宿主环境开辟新线程执行，主线程继续往下执行同步操作；
遇到函数调用，则把它压到 [调用栈 call stack] 中，被压入的函数叫做 [帧 frame]

微任务 (then、async-await) 执行完后，会将其 [回调函数] 入队到 [微任务队列 Microtask Queue] 中；
这些 [回调函数] 会在 [调用栈] 被清空时，**立即**压入 [调用栈] 执行
   -执行回调函数的过程中，注册本次循环的微任务和宏任务；
   -遇到异步操作则开辟新线程执行，主线程继续往下执行同步操作……

宏任务 (Axios、定时器) 执行完后，会将其 [回调函数] 入队到 [消息队列 Message Queue] 中；
调用栈中的代码执行完后，[消息队列] 中的 [回调函数] 会被压入 [调用栈] 中执行
   -执行回调函数的过程中，注册本次循环的微任务和宏任务；
   -遇到异步操作则开辟新线程执行，主线程继续往下执行同步操作……

**总结-执行顺序**：在同一次循环下，同步代码 > 微任务 > 宏任务

> #### 面试题 1

```js
console.log("A");

setTimeout(() => {
    console.log("B");
    new Promise(resolve => {
        console.log("C");
        resolve();
    }).then(() => {
        console.log("D");
    });
}, 0);

new Promise(resolve => {
    console.log("E");
    resolve();
}).then(() => {
    console.log("F");
});

setTimeout(() => {
    console.log("G");

    new Promise(resolve => {
        console.log("H");
        resolve();
    }).then(() => {
        console.log("I");
    });
}, 0);
```

输出顺序：`A` `E` `F` `B` `C` `D` `G` `H` `I`

```
- 主线程中有俩同步执行的 log `A` `E`；
- 然后是微任务 then()：
    - 其回调函数中有同步执行的 log `F`；
    - 没有微任务；
    - 没有宏任务；
- 然后是第一个宏任务 setTimeout：
    - 其回调函数中有同步执行的 log `B` `C`；
    - 然后是微任务：
        - 其回调函数中有同步执行的 log `D`；
        - 没有微任务；
        - 没有宏任务；
    - 没有宏任务；
- 然后是第二个宏任务 setTimeout：
    - 其回调函数中有同步执行的 log `G` `H`；
    - 然后是微任务：
        - 其回调函数中有同步执行的 log `I`；
        - 没有微任务；
        - 没有宏任务；
    - 没有宏任务
```

> #### 面试题 2

```js
document.addEventListener('click', function () {
    Promise.resolve().then(() => console.log(1));
    console.log(2);
})
document.addEventListener('click', function () {
    Promise.resolve().then(() => console.log(3));
    console.log(4);
})
```

输出顺序：`2` `1` `4` `3`

解析：① 事件是异步操作中的宏任务；两个事件的完成时间一样，先注册的先执行，所以会先执行第 1 个事件；
② 在第 1 个事件中，有 then 操作，属于异步操作中的微任务，所以输出顺序是 `2` `1`；
③ 在第 2 个事件中，同理可得输出顺序 `4` `3`

> #### 面试题 3

```js
const fs = require("fs");
const util = require("util");

const myReadFile = util.promisify(fs.readFile);

console.log("A");
myReadFile("./1.txt")
    .then(_ => console.log("B"))
    .then(_ => console.log("C"));
myReadFile("./2.txt").then(_ => console.log("D"));
setTimeout(() => console.log("E"), 0);
console.log("F");
```

输出顺序：`A` `F` `E` `B` `C` `D`  /  `A` `F` `E` `D` `B` `C`

解析：① 先执行同步任务，所以先输出 `A` `F`
② [读取操作] & [事件] 都是宏任务，[事件] 完成的时间较短，先进入任务队列，所以先输出 `E`
③ 先完成的 [读取操作] 先进入任务队列；如果先读取完 1.txt 文件，则先执行该宏任务下的微任务，输出 `B` `C`；否则先输出 `D`

> #### 面试题 4

```js
setTimeout(function () {
    console.log('timer 1');
}, 0);
requestAnimationFrame(function () {
    console.log('UI update');
});
setTimeout(function () {
    console.log('timer 2');
}, 0);
new Promise(function executor(resolve) {
    console.log('promise 1');
    resolve();
    console.log('promise 2');
}).then(function () {
    console.log('promise then');
});
console.log('end');
```

执行顺序1：`promise 1` - `promise 2` - `end` - `promise then` - `timer1` - `timer2` - `UI update`
执行顺序2：`promise 1` - `promise 2` - `end` - `promise then` - `UI update` - `timer1` - `timer2`

解析：① 先执行同步任务：构造函数 Promise 的执行器 executor 是同步执行的
所以前 3 的输出顺序一定是 `promise 1` - `promise 2` - `end`；
② Promise 实例的 then 操作是异步操作中的微任务，而剩下的操作都是异步操作中的宏任务
所以后面一定是先打印 `promise then`；
③ 对于两个 setTimeout 异步操作，因为完成的时间一样，所以是先注册的先执行，所以一定是 `timer1` - `timer2`；
④ 对于 setTimeout 和 requestAnimationFrame：setTimeout 是人为设置的异步操作，requestAnimationFrame 是浏览器自动执行的异步操作，我们每次执行定时器时，都不知道 requestAnimationFrame 执行到哪里，所以有可能是 requestAnimationFrame 先输出，也有可能是 setTimeout 先输出；
∴ 输出的顺序有如上两种情况！！！

另外，requestAnimationFrame 用于浏览器画面的渲染，是按照浏览器的刷新率来执行的，也就是屏幕刷新⼀次 函数就触发⼀次，每秒约执行 60 次；setTimeout 是定时器，默认每秒约执行 200 次；因为 setTimeout 的执行速度较快，所以它先输出的概率较高！！

# 浏览器对 JS 的处理

## 页面加载的流程

1. **创建** document 对象。解析 HTML 及其文本内容后，添加 Element 对象和 Text 节点到文档中
   这个阶段的 **`document.readyState = 'loading'`**

2. 如果遇到 `link` 标签引入外部 CSS 文件，浏览器会创建新线程加载 CSS 文件，同时继续解析文档（异步）

3. ① 如果引入外部 JS 文件，且没有设置 `async`、`defer` 等操作，浏览器会加载 JS 文件，阻塞主线程，等 JS 文件加载、并执行完，再继续解析文档（同步）

   ② 如果引入外部 JS 文件，且设置了 `async`、`defer` 等操作，浏览器会创建新线程加载 JS 文件，同时继续解析文档（异步）
   
4. 如果遇到 `img`、`iframe` 等标签，浏览器会异步加载 `src`，同时继续解析文档（异步）

5. 文档**解析**完成后，设置有 `defer` 的脚本会按顺序执行（同步）、此时的 **`document.readyState = 'interactive'`**

   <u>document 对象触发 `DOMContentLoaded` 事件</u>，这标志着程序从 [同步脚本执行阶段] → [事件驱动阶段]

6. 等所有异步的数据资源**加载**完后，**`document.readyState = 'complete'`**，<u>window 对象触发 `load` 事件</u>

   从此，以异步响应方式处理用户输入、网络事件...

简单来说：
① **创建** document 对象，此时 `document.readyState = 'loading'`
② 文档**解析**完成：此时 `document.readyState = 'interactive'`，document 对象触发 `DOMContentLoaded` 事件
③ 文档**加载**完成：此时 `document.readyState = 'complete'`，window 对象触发 `load` 事件

```js
console.log("create", document.readyState); // create loading

// DOM 解析完成
document.addEventListener(
    "DOMContentLoaded",
    () => console.log("DOMContentLoaded", document.readyState) // DOMContentLoaded interactive
);

// 资源加载完
window.onload = function () {
    console.log("onload", document.readyState); // onload complete
};

// document.readyState 更新
document.onreadystatechange = function () {
    console.log("onreadystatechange", document.readyState);
    // onreadystatechange interactive
    // onreadystatechange complete
};
```

## 同步加载 JS

- 浏览器会同步加载 JS 代码，以防 JS 操作 DOM 时出错

- 浏览器加载 .html 时，如果遇到了 script 标签，会停下来 先把 script 标签里的 JS 执行完
  如果 script 标签里 有外部文件，那就必须等待其下载、执行，然后浏览器才会继续往下执行

  如果网速较慢，JS 文件的下载就会阻止后面代码的执行

## 重构 & 重绘

DOM 树生成后，会等待 CSS 树生成，之后再等 Render 树生成，最后才绘制页面

我们可以简单地理解为：
重构是结构发生了变化，进而影响页面布局；
重绘只是某些样式发生了变化，不会影响布局

> #### reflow 重构

- 页面渲染完成后，若 DOM 树被修改，就会生成新的 DOM 树，再生成新的 Render 树
- 重构的效率较低，尽量减少修改 DOM 的次数
- 会触发重构的情况：① DOM 节点的删除、添加...   ② 改变 DOM 节点的宽高、位置、display: none / block...

> #### repaint 重绘

- 改变背景颜色、文字颜色等，导致 CSS 树的改变，叫重绘

- 重绘只影响页面的一部分，不会导致整个页面重构，影响比较小

  （如果改变的是文字大小，就会触发重构）

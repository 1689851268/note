# 异步执行原理

## 单线程的 JS

-   单线程执行：同一时间只能执行一个任务，只有把当前任务完成，才能执行下一个任务
    这会造成**线程阻塞**（当前任务耗时过长，导致后面的任务不能被执行）

```js
for (let i = 0; i < 5000; i++) {
    console.log("一个耗时很长的同步任务");
}
console.log("后面的同步任务");
```

上例就出现了线程阻塞：`for` 循环执行完后，才会执行后面的代码

<br>

## 多线程的浏览器

-   浏览器是多线程的，当 JS 需要执行异步任务时，浏览器会另外启动一个线程去执行该任务
-   JS 是单线程的，指的是执行 JS 代码的线程只有一个，是浏览器提供的 JS 引擎线程 (主线程)
-   除此之外，浏览器中还有定时器线程、异步 HTTP 请求线程等... 这些线程主要不是用来执行 JS 代码的

eg：主线程中需要发送数据请求，浏览器就会把这个任务交给异步 HTTP 请求线程去执行，等请求数据返回后，再将 callback 里需要执行的 JS 回调交给 JS 引擎线程去执行。也就是说，浏览器才是真正执行发送请求这个任务的角色，而 JS 只是负责执行最后的回调处理。所以这里的异步不是 JS 自身实现的，而是浏览器为其提供的能力

# 同步 & 异步

## 同步任务 & 异步任务

-   同步任务：即 “非耗时程序”，都是在 [主线程] 上排队执行的任务
-   异步任务：即 “耗时程序” (**事件**、**Ajax**、**定时器**等)，由 JS 委托宿主环境 (Node / 浏览器) 执行
-   JS 的执行顺序遵循单线程的异步模型：**同步在前，异步在后**
-   根据 JS 单线程异步模型的规则，异步操作得通过 [回调函数] 实现
    ∴ 异步任务一定有回调函数！但请注意，有回调函数的不一定是异步任务！

```js
setTimeout(function () {
    console.log("异步任务");
}, 0);

[1].forEach(item => {
    console.log("同步任务");
});
```

在 [主线程] 中，JS 代码会被**从上往下**地一行行执行；遇到异步任务，则委托给宿主环境开辟新线程执行；
异步任务执行完后，会将其 [回调函数] 放入任务队列 Event Queue 等待；
等主线程将同步任务都执行完，就从 Event Queue 中获取最先入队的 [回调函数] 执行

在事件驱动的模式下，至少包含了一个执行循环来检测任务队列中是否有新任务。
通过不断循环，去取出异步任务的回调来执行，这个过程就是事件循环，每一次循环就是一个事件周期

```js
setTimeout(() => {
    console.log("异步任务");
}, 0);
for (let i = 0; i < 5000; i++) {
    console.log("一个耗时很长的同步任务");
}
// for 循环执行完后，才会执行 setTimeout 里面的代码
```

> #### 如果出现多个异步操作：

在异步操作完成时间**不同**的情况下，先完成的异步操作会先进入 Event Queue：
下例输出顺序：`sync` - `setTimeout 2` - `setTimeout 1`

```js
setTimeout(function () {
    // 这个耗时 2s
    console.log("setTimeout 1");
}, 2000);

setTimeout(function () {
    // 这个耗时 1s
    console.log("setTimeout 2");
}, 1000);

console.log("sync");
```

在异步操作完成时间**相同**的情况下，先**注册**的异步操作会先进入 Event Queue：
下例输出顺序：`sync` - `setTimeout 1` - `setTimeout 2`

```js
setTimeout(function () {
    // 前面的代码先注册
    console.log("setTimeout 1");
}, 2000);

setTimeout(function () {
    // 后面的代码后注册
    console.log("setTimeout 2");
}, 2000);

console.log("sync");
```

任务队列前面的异步操作被阻塞时，后面的异步操作也会被阻塞：
下例的输出顺序：`sync` - `setTimeout 1 start` - `setTimeout 1 end` - `setTimeout 2`

```js
setTimeout(function () {
    // 先进入执行栈
    console.log("setTimeout 1 start");
    for (let i = 0; i < 5000; i++) {
        // 被阻塞了
        console.log("一个耗时很长的同步任务");
    }
    console.log("setTimeout 1 end");
}, 1000);

setTimeout(function () {
    // 后面的异步操作也会被阻塞
    console.log("setTimeout 2");
}, 1000);
console.log("sync");
```

## 宏任务 & 微任务

> #### 宏任务 (macro task)

宏任务是 JS 中最原始的异步任务，包括：定时器、Ajax...
宏任务需要特定的异步线程去执行，有明确的异步任务去执行，有回调

常见的宏任务：① **事件**、② **定时器**、③ **Ajax**、④ **I/O 操作**、⑤ **`requestAnimationFrame`**(浏览器) ...
有些地方会说 UI Rendering 也是宏任务，但这是和微任务平行的操作，应该算是微任务；
`requestAnimationFrame` 是页面重绘前执行的操作（⑥ **重绘** 也是⼀个宏任务）

> #### 微任务 (micro task)

微任务是随着 ECMA 标准升级提出的新的异步任务
微任务不需要特定的异步线程去执行，没有明确的异步任务去执行，只有回调

常见的微任务：① **`process.nextTick`**(node.js)、② **`MutationObserver`**(浏览器)、③ **`Promise实例.then catch finally`**、**`async-await`**...

<br>

## 事件循环

<img src="picture/43.%20JS%20%E7%9A%84%E6%89%A7%E8%A1%8C/image-20221030153848187-1667115531930-1.png" alt="image-20221030153848187" style="zoom:50%;" />

[主线程] 会从上往下一行一行地执行 JS 代码；并注册本次循环的微任务和宏任务
遇到异步操作，则委托给宿主环境开辟新线程执行，主线程继续往下执行同步操作；
遇到函数调用，则把它压到 [调用栈 call stack] 中，被压入的函数叫做 [帧 frame]

宏任务 (Ajax、定时器) 执行完后，会将其 [回调函数] 入队到 [宏任务队列 Macrotask Queue] 中
微任务 (Promise.then、async-await) 执行完后，会将其 [回调函数] 入队到 [微任务队列 Microtask Queue] 中

1. JS 引擎首先从 Macrotask Queue 中取出第 1 个任务
2. 执行完毕后，再将 Microtask Queue 中所有的任务取出，按照顺序分别全部执行（这里不仅包括刚开始执行时队列里的微任务），如果在这一步过程中产生新的微任务，也需要执行，也就是说在执行微任务过程中产生的新的微任务并不会推迟到下一个循环中执行，而是在当前的循环中继续执行
3. 然后再从 Macrotask Queue 中取出下一个任务，执行完毕后，再次将 Microtask Queue 中的全部取出，循环往复，直到两个 Queue 中的任务都取完

**总结-执行顺序**：在同一次循环下，同步代码 > 微任务 > 宏任务

> #### 面试题 1

```js
console.log("A");

setTimeout(() => {
    console.log("B");
    new Promise(resolve => {
        console.log("C");
        resolve();
    }).then(() => {
        console.log("D");
    });
}, 0);

new Promise(resolve => {
    console.log("E");
    resolve();
}).then(() => {
    console.log("F");
    setTimeout(() => {
        console.log("G");
    }, 0);
});
```

输出顺序：`A` `E` `F` `B` `C` `D` `G`

```
- 主线程中有俩同步执行的 log `A` `E`；
- 然后是微任务 then()：
    - 其回调函数中有同步执行的 log `F`；
    - 没有微任务；
    - 有宏任务 setTimeout，异步执行完后，将其 [回调函数] 放置 Macrotask Queue 的最后
- 然后是第 1 个宏任务 setTimeout：
    - 其回调函数中有同步执行的 log `B` `C`；
    - 然后是微任务：
        - 其回调函数中有同步执行的 log `D`；
        - 没有微任务；
        - 没有宏任务；
    - 没有宏任务；
- 然后是第 2 个宏任务 setTimeout：
    - 其回调函数中有同步执行的 log `G`；
    - 没有微任务；
    - 没有宏任务
```

> #### 面试题 2

```js
document.addEventListener("click", function () {
    Promise.resolve().then(() => console.log(1));
    console.log(2);
});
document.addEventListener("click", function () {
    Promise.resolve().then(() => console.log(3));
    console.log(4);
});
```

输出顺序：`2` `1` `4` `3`

解析：① 事件是异步操作中的宏任务；两个事件的完成时间一样，先注册的先执行，所以会先执行第 1 个事件；
② 在第 1 个事件中，有 then 操作，属于异步操作中的微任务，所以输出顺序是 `2` `1`；
③ 在第 2 个事件中，同理可得输出顺序 `4` `3`

> #### 面试题 3

```js
const fs = require("fs");
const util = require("util");

const myReadFile = util.promisify(fs.readFile);

console.log("A");
myReadFile("./1.txt")
    .then(_ => console.log("B"))
    .then(_ => console.log("C"));
myReadFile("./2.txt").then(_ => console.log("D"));
setTimeout(() => console.log("E"), 0);
console.log("F");
```

输出顺序：`A` `F` `E` `B` `C` `D` / `A` `F` `E` `D` `B` `C`

解析：① 先执行同步任务，所以先输出 `A` `F`
② [读取操作] & [事件] 都是宏任务，[事件] 完成的时间较短，先进入任务队列，所以先输出 `E`
③ 先完成的 [读取操作] 先进入任务队列；如果先读取完 1.txt 文件，则先执行该宏任务下的微任务，输出 `B` `C`；否则先输出 `D`

> #### 面试题 4

```js
setTimeout(function () {
    console.log("timer 1");
}, 0);
requestAnimationFrame(function () {
    console.log("UI update");
});
setTimeout(function () {
    console.log("timer 2");
}, 0);
new Promise(function executor(resolve) {
    console.log("promise 1");
    resolve();
    console.log("promise 2");
}).then(function () {
    console.log("promise then");
});
console.log("end");
```

执行顺序 1：`promise 1` - `promise 2` - `end` - `promise then` - `timer1` - `timer2` - `UI update`
执行顺序 2：`promise 1` - `promise 2` - `end` - `promise then` - `UI update` - `timer1` - `timer2`

解析：① 先执行同步任务：构造函数 Promise 的执行器 executor 是同步执行的
所以前 3 的输出顺序一定是 `promise 1` - `promise 2` - `end`；
② Promise 实例的 then 操作是异步操作中的微任务，而剩下的操作都是异步操作中的宏任务
所以后面一定是先打印 `promise then`；
③ 对于两个 setTimeout 异步操作，因为完成的时间一样，所以是先注册的先执行，所以一定是 `timer1` - `timer2`；
④ 对于 setTimeout 和 requestAnimationFrame：setTimeout 是人为设置的异步操作，requestAnimationFrame 是浏览器自动执行的异步操作，我们每次执行定时器时，都不知道 requestAnimationFrame 执行到哪里，所以有可能是 requestAnimationFrame 先输出，也有可能是 setTimeout 先输出；
∴ 输出的顺序有如上两种情况！！！

另外，requestAnimationFrame 用于浏览器画面的渲染，是按照浏览器的刷新率来执行的，也就是屏幕刷新⼀次 函数就触发⼀次，每秒约执行 60 次；setTimeout 是定时器，默认每秒约执行 200 次；因为 setTimeout 的执行速度较快，所以它先输出的概率较高！！

# Node 的事件循环

当 Node.js 启动时，它会初始化一个事件循环，来处理输入的脚本，这个脚本可能调用 `process.nextTick()`、调度计时器、调用异步 API，然后开始处理事件循环

JS 和 Node 是基于 V8 引擎的，浏览器中包含的异步方式在 Node 中也是一样的。除此之外，Node 中还有一些其他的异步形式：
① **文件 I / O**：异步加载本地文件
② **`setImmediate`**：与 `setTimeout` 设置 `0ms` 类似
③ **`process.nextTick`**：在某些同步任务完成后立马执行
④ **`server.close`**、**`socket.on('close', ……)`** 等：关闭回调
这些异步任务的执行就需要依靠 Node 的事件循环机制了

Node 中的 Event Loop 和浏览器中的 Event Loop 是完全不相同的东西。Node 使用 V8 作为 js 的解析引擎，而 I/O 处理方面则使用自己设计的 **libuv**。libuv 是一个基于事件驱动的跨平台抽象层，封装了不同操作系统一些底层特性，对外提供统一的 API，事件循环机制也是它里面的实现的，如下图所示：
<img src="picture/43.%20JS%20%E7%9A%84%E6%89%A7%E8%A1%8C/image-20221030164717290-1667119639504-3.png" alt="image-20221030164717290" style="zoom:50%;" />

根据上图，可以看到 Node 的运行机制如下：
① V8 引擎负责解析 JS 脚本
② 解析后的代码，调用 Node API
③ libuv 库负责 Node API 的执行；
    **libuv 将不同的任务分配给不同的线程，形成一个 Event Loop，以异步的方式将任务的执行结果返回给 V8 引擎**
④ V8 引擎将结果返回给用户

## 事件循环的流程

libuv 引擎中的事件循环分为 6 个阶段，它们会按顺序反复运行。每当进入某个阶段的时候，都会从对应的回调队列中取出函数执行。当队列为空或者执行的回调函数数量到达系统设定的阈值，就会进入下一阶段。下面是事件循环 Event Loop 的流程
<img src="picture/43.%20JS%20%E7%9A%84%E6%89%A7%E8%A1%8C/image-20221030165002308-1667119803751-5.png" alt="image-20221030165002308" style="zoom: 33%;" />

1. **timers 阶段**：执行 timer (`setTimeout`、`setInterval`) 的回调，由 poll 阶段控制
2. **I/O callbacks 阶段**：主要执行系统级别的回调函数，比如 TCP 连接失败的回调
3. **idle, prepare 阶段**：仅 Node 内部使用，可以忽略
4. **poll 阶段**：轮询等待新的链接和请求等事件，执行 I/O 回调等
5. **check 阶段**：执行 `setImmediate` 的回调
6. **close callbacks 阶段**：执行关闭请求的回调函数，比如 `socket.on('close', ...)`

**注意：上面每个阶段都会去执行完当前阶段的任务队列，然后继续执行当前阶段的微任务队列，只有当前阶段所有微任务都执行完了，才会进入下个阶段，这里也是与浏览器中逻辑差异较大的地方**

其中，这里面比较重要的就是第 4 阶段：**poll**，这一阶段中，系统主要做两件事：
① 回到 timer 阶段执行回调
② 执行 I/O 回调

在进入 poll 阶段时如果没有设定 timer 的话，会出现以下情况：
（1）如果 poll 队列不为空，会遍历回调队列并同步执行，直到队列为空或者达到系统限制；
（2）如果 poll 队列为空时，会出现以下情况：
① 如果有 setImmediate 回调需要执行，poll 阶段会停止并且进入到 check 阶段执行回调；
② 如果没有 setImmediate 回调需要执行，会等待回调被加入到队列中并立即执行回调。这里同样会设置个超时时间防止一直等待
当设定了 timer 且 poll 队列为空，则会判断是否有 timer 超时，如果有的就会回到 timer 阶段执行回调

## 宏任务 & 微任务

常见的宏任务：`setTimeout`、`setInterval`、`setImmediate`、`script`(整体代码)、I/O 操作...
常见的微任务：`process.nextTick`、`new Promise().then`...

`process.nextTick()` 是 node 中新引入的一个任务队列，它会在上述各个阶段结束时，在进入下一个阶段之前立即执行

```js
setTimeout(() => {
    console.log("timeout");
}, 0);

Promise.resolve().then(() => {
    console.error("promise");
});

process.nextTick(() => {
    console.error("nextTick");
});
```

输出结果：`nextTick` - `promise` - `timeout`；可以看到，`process.nextTick()` 优先于 `promise 的回调` 执行

> #### `setImmediate` 和 `setTimeout`

setImmediate 和 setTimeout 很相似，主要区别在于调用的时机不同：
setImmediate：在 poll 阶段完成时执行，即 check 阶段；
setTimeout：在 poll 阶段为空闲时，且设定时间到达后执行，但它在 timer 阶段执行；

```js
setTimeout(() => {
    console.log("timeout");
}, 0);

setImmediate(() => {
    console.log("setImmediate");
});
```

输出结果：`timeout` - `setImmediate`

在上面代码的执行过程中，第一轮循环后，分别将 `setTimeout` 和 `setImmediate` 加入了各自阶段的任务队列。第二轮循环首先进入 timers 阶段，执行定时器队列回调，然后 pending callbacks 和 poll 阶段没有任务，因此进入 check 阶段执行 `setImmediate` 回调。所以最后输出为 `timeout`、`setImmediate`

## Node 与浏览器事件循环的差异

-   Node.js：microtask 在事件循环的各个阶段之间执行
-   浏览器：microtask 在事件循环的 macrotask 执行完之后执行

<img src="picture/43.%20JS%20%E7%9A%84%E6%89%A7%E8%A1%8C/image-20221030214003764-1667137205605-9.png" alt="image-20221030214003764" style="zoom:50%;" />

> #### Nodejs 和浏览器的事件循环流程对比

1. 执行全局的 Script 代码（与浏览器无差）
2. 把微任务队列清空：注意，Node 清空微任务队列的手法比较特别。在浏览器中，我们只有一个微任务队列需要接受处理；但在 Node 中，有两类微任务队列：next-tick 队列和其它队列。其中这个 next-tick 队列，专门用来收敛 process.nextTick 派发的异步任务。在清空队列时，优先清空 next-tick 队列中的任务，随后才会清空其它微任务
3. 开始执行 macro-task（宏任务）。注意，Node 执行宏任务的方式与浏览器不同：在浏览器中，我们每次出队并执行一个宏任务；而在 Node 中，我们每次会尝试清空当前阶段对应宏任务队列里的所有任务（除非达到系统限制）
4. 步骤 3 开始，会进入 3 - 2 - 3 - 2 … 的循环
